## Local Setup

### Step 1: Python environment

This project requires **Python 3.12.x**. Python 3.13 is not supported because
`tensorflow==2.16.2` does not publish wheels for it.

**Windows:**

```powershell
# Verify you have Python 3.12 installed
py -3.12 --version

# Create isolated virtual environment using Python 3.12
py -3.12 -m venv .venv
.venv\Scripts\activate

# Verify activation (you should see (.venv) in your prompt)
where python
```

**Linux / macOS:**

```bash
# Verify Python 3.12 is installed
python3.12 --version

# Create isolated virtual environment
python3.12 -m venv .venv
source .venv/bin/activate

# Verify activation
which python   # Should show .venv path
```

### Step 2: Install dependencies

```bash
# Option A (recommended for development): editable install with dev extras
# This installs the project in editable mode so code changes take effect immediately.
pip install -e ".[dev]"

# Option B: install from requirements files directly
pip install -r requirements.txt        # production dependencies
pip install -r requirements-dev.txt    # adds pytest, black, ruff
```

### Step 3: Configure environment (optional)

```bash
# Copy the example env file
cp .env.example .env

# Edit if you want to change log level or data directory
```

### Step 4: Verify the installation

```bash
# Run the test suite — all 145 should pass
pytest tests/ -q

# Launch the app (models don't need to be trained — the UI handles missing models gracefully)
streamlit run streamlit_app/app.py
```

## Project Structure Quick Reference

```
src/                    ← Pure Python backend — no Streamlit imports
streamlit_app/          ← Frontend only — no ML logic here
config/                 ← Single source of truth for all settings
tests/                  ← pytest suite — mirrors src/ structure
docs/                   ← Project documentation
models/saved/           ← .keras files generated by training
performance_plots/      ← PNGs and JSON artifacts from evaluation
logs/                   ← Application logs (gitignored)
```

**Strict boundary:** `src/` modules must never import from `streamlit_app/`.
The dependency always flows one way: `streamlit_app/` → `src/`.

## Adding a New Model

This is the most common extension task. Here is the exact 4-step process:

### Step 1: Create the model file

```python
# src/models/my_new_model.py

"""
My New Model — DigitVision.

Why it exists:
    Describe the architectural idea and what it demonstrates.
"""

from __future__ import annotations

import tensorflow as tf


def build_my_new_model(learning_rate: float = 0.001) -> tf.keras.Model:
    """
    Build and compile My New Model.

    Architecture:
        Describe layers and design choices here.

    Args:
        learning_rate: Adam optimiser learning rate.

    Returns:
        Compiled tf.keras.Model ready for training.
    """
    inputs = tf.keras.layers.Input(shape=(28, 28, 1), name="input")

    # ... your layers here ...

    outputs = tf.keras.layers.Dense(10, activation="softmax", name="predictions")(x)
    model = tf.keras.Model(inputs=inputs, outputs=outputs, name="my_new_model")

    model.compile(
        optimizer=tf.keras.optimizers.Adam(learning_rate=learning_rate),
        loss="categorical_crossentropy",
        metrics=["accuracy"],
    )
    return model
```

### Step 2: Register the model

```python
# src/models/__init__.py

from src.models.my_new_model import build_my_new_model  # add this import

_MODEL_REGISTRY: dict[str, Callable] = {
    "dense_nn": build_dense_nn,
    "lenet5": build_lenet5,
    "custom_cnn": build_custom_cnn,
    "my_new_model": build_my_new_model,  # add this entry
}
```

### Step 3: Register paths and display names

```python
# config/config.py

MODEL_DISPLAY_NAMES: dict[str, str] = {
    ...
    "my_new_model": "My New Model",  # add this
}

MODEL_PATHS: dict[str, Path] = {
    ...
    "my_new_model": MODELS_DIR / "my_new_model.keras",  # add this
}

MODEL_TRAINING_CONFIG: dict[str, dict[str, Any]] = {
    ...
    "my_new_model": {
        "epochs": 20,
        "batch_size": 128,
        "learning_rate": 0.001,
    },
}
```

### Step 4: Add tests

```python
# tests/test_models.py

class TestMyNewModel:
    @pytest.fixture(autouse=True)
    def build(self) -> None:
        self.model = build_model("my_new_model")

    def test_builds_successfully(self) -> None:
        assert isinstance(self.model, tf.keras.Model)

    def test_output_shape_is_10_classes(self) -> None:
        output = self.model.predict(np.zeros((1, 28, 28, 1), dtype=np.float32), verbose=0)
        assert output.shape == (1, 10)

    def test_parameter_count_in_expected_range(self) -> None:
        n_params = self.model.count_params()
        assert 10_000 <= n_params <= 500_000  # set appropriate bounds
```

That's it. The training script, evaluation script, and Streamlit comparison playground
all automatically pick up the new model by reading from the registry.

## Running Tests

```bash
# Full suite
pytest tests/ -v

# With coverage
pytest tests/ --cov=src --cov-report=term-missing

# Single file
pytest tests/test_preprocessing.py -v

# Single test
pytest tests/test_models.py::TestCustomCNN::test_has_global_average_pooling -v

# Skip slow tests (model training)
pytest tests/ -m "not slow"

# Integration tests only (require trained models)
pytest tests/ -m "integration"

# Stop after first failure
pytest tests/ -x
```

### Test Architecture

All test fixtures are in `tests/conftest.py`. Tests are designed to be:

- **Fast** — No MNIST download, no real training. Synthetic numpy arrays and stub Keras models.
- **Offline** — No network calls during testing.
- **Isolated** — Each test gets a fresh `tmp_path` for file I/O.
- **Deterministic** — All random generators use fixed seeds.

## Debugging

### Common issues and solutions

**"ModuleNotFoundError: No module named 'src'"**
```bash
# Run from the project root, not from inside src/
cd "path/to/Handwritten Digit Recognizer"
python -m src.train --model custom_cnn
```

**"FileNotFoundError: Model has not been trained yet"**
```bash
# Train the model first
python -m src.train --model custom_cnn
# Then run evaluation or launch the app
```

**Streamlit app loads but shows 'No model found'**
- Models must be trained before the dashboard shows evaluation data
- Run `make train && make evaluate`
- The app gracefully shows placeholder content when models are absent

**Grad-CAM returns None for a model**
- Only models with Conv2D layers support Grad-CAM
- Dense NN always returns None — this is correct and expected behaviour

**Canvas prediction feels wrong**
- Check that the canvas is configured with `background_color="#000000"` (black)
- The preprocessing pipeline inverts the alpha channel, expecting a black canvas background

### Debugging with logging

```bash
# Enable DEBUG-level logging
export DIGITVISION_LOG_LEVEL=DEBUG

# Or on Windows PowerShell
$env:DIGITVISION_LOG_LEVEL = "DEBUG"

streamlit run streamlit_app/app.py
```

Logs are written to both `logs/digitvision.log` and the terminal.

## Logging

DigitVision uses Python's built-in `logging` module with a centralised configuration in `src/logger.py`.

### Log levels

| Level | Usage |
|-------|-------|
| `DEBUG` | Preprocessing steps, model layer details |
| `INFO` | Training start/end, metric values, file saves |
| `WARNING` | Unexpected but non-fatal conditions |
| `ERROR` | Failed file operations, model loading errors |

### Using the logger in new modules

```python
from src.logger import get_logger

logger = get_logger(__name__)

def my_function() -> None:
    logger.info("Starting my_function")
    logger.debug("Input shape: %s", input.shape)
```

The `%s` formatting style is used instead of f-strings in log calls.
This defers string formatting until the log level check passes, saving CPU cycles
when DEBUG logging is disabled in production.

## Docker

### Building and running

```bash
# Build the image
docker compose build

# Run the application
docker compose up

# Run in background
docker compose up -d

# Stop
docker compose down

# View logs
docker compose logs -f
```

### Dockerfile structure

The Dockerfile uses a **single-stage build** from `python:3.12-slim`:

1. Install system libraries required by OpenCV and TensorFlow (`libglib2.0-0`, `libgomp1`, etc.)
2. Copy `requirements.txt` and install Python dependencies (this layer is cached by Docker as long as `requirements.txt` does not change)
3. Copy application source
4. Create required runtime directories and switch to a non-root user

```bash
# Build directly with Docker (no compose)
docker build -t digitvision:latest .
docker run -p 8501:8501 digitvision:latest
```

### Volume mounting

The `docker-compose.yml` mounts four host directories for data persistence:

```yaml
volumes:
  - ./models:/app/models
  - ./data:/app/data
  - ./performance_plots:/app/performance_plots
  - ./logs:/app/logs
```

This means models trained on the host are immediately visible inside the container,
and artifacts generated during evaluation are saved back to the host.

## CI/CD Pipeline

GitHub Actions runs two jobs on every push and pull request to `main`:

```
push to main
│
├── Job 1: Lint & Format Check (ubuntu-latest)
│   ├── ruff check src/ tests/ config/
│   └── black --check src/ tests/ config/
│
└── Job 2: Run Tests (ubuntu-latest) — only if lint passes
    ├── pip install -r requirements-dev.txt
    ├── pytest tests/ -m "not slow" --cov=src
    └── Upload coverage.xml as artifact
```

### Running CI checks locally before pushing

```bash
# Run exactly what CI runs
make lint    # ruff + black check
make test    # pytest
```

### Adding a new CI check

Edit `.github/workflows/ci.yml`. The workflow uses GitHub Actions caching for pip
packages, so dependency installation is fast on subsequent runs.

## Code Quality Standards

DigitVision follows strict code quality rules enforced by CI:

### Formatting: Black

```bash
# Check (CI mode — fails if any file would be reformatted)
black --check src/ tests/ config/

# Auto-format (developer mode)
black src/ tests/ config/
```

Black is configured in `pyproject.toml` with `line-length = 100`.

### Linting: Ruff

```bash
ruff check src/ tests/ config/
ruff check --fix src/ tests/ config/   # auto-fix where possible
```

### Type hints

Every public function must have type-annotated parameters and return type.
`TYPE_CHECKING` blocks are used for imports that would cause circular dependencies.

### Docstrings

All public functions use Google-style docstrings with at minimum:
- A one-line summary
- `Args:` block
- `Returns:` block
- `Raises:` block (if applicable)

The docstring explains **why** the function exists, not just what it does.
